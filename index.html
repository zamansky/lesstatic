  1. What and Why

    Why another static site generator? Well, I've been keeping my personal blog C'est la Z up using GitHub pages which support Jekyll. Jekyll has a lot of nice things going for it. It's simple, pretty easy and quick. There were a couple of things I didn't like. It was a little less flexible than I'd like which wasn't a big deal and it was in Ruby. Ruby's all fine and good, but I'm more of a Python guy and really don't any desire to add another similar language to my vocabulary at this time but I also feel helpless every time I get a weird error.

    I looked at the python offerings and they were either too simple and restrictive, too comples, or just didn't speak to me. So, there you have it.

    My hope is that LesStatic has a simpler configuration and use patterns than the more complex generators but be more flexible than the drop dead simple ones. In short, something that hits the sweet spot for me.

  2. Features
    • Simple configuration file driven directory structure
    • Markdown and html processing
    • Pygments for code (see notes)
    • Jinja2 templates
    • Written in Python
  3. Quickstart
    1. Download or clone from here
    2. cd into the lesstatic directory
    3. sudo pip install .
    4. create a new project with template files: lesstatic --init some_new_directory_name
    5. switch into that directory: cd some_new_directory_name
    6. start the building server: lesstatic --serve
    7. open a browser window to http://localhost:8000
    8. Edit the files as needed
    9. Ctrl-c to break out of the server
    10. Also look at the demosite folder in the repository
  4. Installation
    1. Download or clone from here
    2. sudo pip install .
    3. for testing, you can run it out of it's own directory (python ./lesstatic, for instance)
  5. Usage

    You create your site in a directory or folder that will hold subdirectories with your projects various components. The actual structure is described in the configuration file named config.yaml located in the toplevel of the project directory. The configuration file and directory structure are described lated in this document. You can use the --init parameter to lesstatic to create a skeleton project from which you can work.

    Command Line Arguments:

    ArgumentDescription
    -h, --helpPrints a help message
    -i, --initcreates a skeleton project. FOLDER must be a new directory name or you must be in an empty directory
    -p, --portRuns the server on the specified port (overrides config.yaml)
    -s, --serveRun the server (which will rebuild the project on changes)
    Running with no arguments tries to rebuild the web site in the current project folder. Running with only the folder name tries to build the site in that folder.
  6. Project directory structure

    7. Technically, all you need is a folder with a single file config.yaml which will tell lesstatic where everything else lives. The file is a yaml file and described in the next section. Running lessatatic with the --init option will create a sample config.yaml and directory structure.

  7. Configuration

    All configuration is specified in the config.yaml file in the toplevel project directory. It's a standard yaml file consisting of key:value pairs or in the case of the extensions option, a list of key:value pairs.

    Options:

    OptionvalueNotes
    portvalid port numberThe port to run the server on
    templatesfolder nameThe folder where you put the jinja2 templates
    contentfolder nameThe folder where you put your pages
    staticfolder namestatic stuff to be copied to the site without processing
    sitefolder namedestination folder for the built site
    static_destfolder namefolder under the site destination folder where the static stuff will be copied
    extensionsa list of valid filename extensions and what processor to usecurrently only markdown and html are supported, a key value pair might be md: markdown or html: html
  8. Template Files

    Lesstatic uses Jinja2 as its templating system. Just place all your templates in this folder.

  9. Content files can be either markdown or html with a yaml block on the top. Certain yaml variables, must be defined in the top matter, they are:

    a template filenamethe name of a jinja2 block specified in the above template
    keyvalueNotes
    layoutThe template to use
    blockWhere to put the processed page in the template
    Everything else specified in the yaml top section is a variable (which can be a list or dictionary - see the --init based sample). You can use these as substitutions in your markdown or html file.
    • Pygments should work for html files but you might need to copy in a custom css file into your static folder.
    • Pygments with markdown will certainly need css files copied into your static folders.
